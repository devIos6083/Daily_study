## 1. Git 핵심 명령어

### clone

GitHub 같은 리모트 리포지토리를 **내 컴퓨터에 통째로 복사**하는 명령어.  
단순한 파일 복사가 아니라 커밋 이력과 브랜치 정보까지 전부 가져온다.  
새 프로젝트에 참가할 때 맨 처음 사용한다.

GitHubのようなリモートリポジトリを**自分(じぶん)のパソコンにまるごとコピー**する命令語(めいれいご)。

```bash
git clone https://github.com/user/repo.git
```

---

### branch

메인 흐름에서 벗어나 **독립적으로 작업할 수 있는 공간**을 만드는 명령어.  
메인 코드에 영향을 주지 않고 안전하게 기능 개발이나 버그 수정을 진행할 수 있다.

メインの流(なが)れから離(はな)れて**独立的(どくりつてき)に作業(さぎょう)できる空間(くうかん)**を作(つく)る命令語(めいれいご)。

```bash
git branch feature-login     # 브랜치 생성
git branch                   # 브랜치 목록 확인
git branch -d feature-login  # 브랜치 삭제
```

---

### checkout

**작업할 브랜치를 전환**하는 명령어.  
브랜치 이동뿐만 아니라 특정 커밋 시점으로도 이동할 수 있다.  
최근에는 `git switch`와 `git restore`로 역할이 분리되었다.

**作業(さぎょう)するブランチを切(き)り替(か)える**命令語(めいれいご)。

```bash
git checkout feature-login   # 브랜치 이동
git switch feature-login     # 최신 방식 (동일한 역할)
```

---

### log

커밋 이력을 **시간순으로 보여주는** 명령어.  
누가, 언제, 무엇을 변경했는지 한눈에 파악할 수 있다.

コミットの履歴(りれき)を**時間順(じかんじゅん)に見(み)せてくれる**命令語(めいれいご)。

```bash
git log                 # 전체 이력
git log --oneline       # 한 줄 요약
git log --oneline --graph  # 브랜치 흐름 시각화
```

---

### status

현재 **워크트리와 스테이지의 상태**를 확인하는 명령어.  
어떤 파일이 수정됐는지, 스테이지에 올라갔는지, 아직 커밋되지 않은 변경이 있는지 파악할 수 있다.  
작업 중에 **습관적으로 자주 확인**하는 명령어.

現在(げんざい)の**ワークツリーとステージの状態(じょうたい)**を確認(かくにん)する命令語(めいれいご)。

```bash
git status
```

---

### add

수정한 파일을 **스테이지에 올리는** 명령어.  
변경한 파일 전부가 아니라 **원하는 파일만 골라서** 올릴 수 있어 커밋 내용을 세밀하게 조정할 수 있다.

修正(しゅうせい)したファイルを**ステージに上(あ)げる**命令語(めいれいご)。

```bash
git add 파일명          # 특정 파일만
git add .              # 전체 파일
```

---

### commit

스테이지에 올라간 변경사항을 **로컬 리포지토리에 기록**하는 명령어.  
커밋은 "저장"이 아니라 **"의미 있는 작업 단위의 기록"**이다.  
메시지는 무엇을, 왜 변경했는지 명확하게 작성해야 한다.

ステージに上(あ)がった変更事項(へんこうじこう)を**ローカルリポジトリに記録(きろく)**する命令語(めいれいご)。

```bash
git commit -m "feat: 로그인 기능 추가"
```

---

### diff

파일의 현재 상태와 이전 상태를 비교해서 **어떤 부분이 바뀌었는지** 보여주는 명령어.  
커밋 전에 변경 내용을 최종 확인할 때 유용하다.

ファイルの現在(げんざい)の状態(じょうたい)と以前(いぜん)の状態(じょうたい)を比較(ひかく)して**どの部分(ぶぶん)が変(か)わったか**を見(み)せてくれる命令語(めいれいご)。

```bash
git diff             # 워크트리 vs 스테이지
git diff --staged    # 스테이지 vs 마지막 커밋
```

---

### push

로컬에 쌓인 커밋을 **리모트 리포지토리에 업로드**하는 명령어.  
push해야 비로소 팀원이 변경사항을 볼 수 있다.

ローカルに積(つ)み上(あ)げたコミットを**リモートリポジトリにアップロード**する命令語(めいれいご)。

```bash
git push origin feature-login
```

---

### pull

리모트 리포지토리의 최신 내용을 가져와서 **현재 브랜치에 자동으로 반영**하는 명령어.  
내부적으로 `fetch + merge`가 동시에 이루어지므로 충돌(conflict)이 발생할 수 있다.

リモートリポジトリの最新内容(さいしんないよう)を持(も)ってきて**現在(げんざい)のブランチに自動(じどう)で反映(はんえい)**する命令語(めいれいご)。

```bash
git pull origin main
# 내부적으로: fetch + merge 동시 실행
```

---

### fetch

리모트 리포지토리의 변경 내용을 **가져오기만 하고 반영하지 않는** 명령어.  
pull과 달리 merge가 자동으로 이루어지지 않아 내용을 확인한 후 직접 merge 여부를 결정할 수 있다.

リモートリポジトリの変更内容(へんこうないよう)を**持(も)ってくるだけで反映(はんえい)しない**命令語(めいれいご)。

```bash
git fetch origin     # 가져오기만
git merge origin/main  # 확인 후 직접 merge
```

---

### merge

**다른 브랜치의 변경사항을 현재 브랜치에 합치는** 명령어.  
충돌이 없으면 자동으로 머지되고, 충돌이 발생하면 수동으로 해결해야 한다.

**他(ほか)のブランチの変更事項(へんこうじこう)を現在(げんざい)のブランチに合(あ)わせる**命令語(めいれいご)。

```bash
git merge feature-login   # feature-login을 현재 브랜치에 합치기
```

## 2. Pull Request란?

자신이 작업한 브랜치의 변경사항을 메인 브랜치에 합치기 전에  
**팀원에게 검토를 요청하는 협업 과정**.

自分(じぶん)が作業(さぎょう)したブランチの変更事項(へんこうじこう)をメインブランチに合(あ)わせる前(まえ)に  
**チームメンバーに検討(けんとう)を依頼(いらい)する協力(きょうりょく)の過程(かてい)**。

```
코드를 직접 merge하는 것이 아니라
"이 변경사항을 반영해도 될까요?" 를 사전에 확인받는 것이 핵심!
```

### 왜 필요한가?

```
혼자 개발하는 경우는 드물다.
팀 단위로 개발할 때 변경사항이:
  ✅ 기존 기능과 문제가 없는지
  ✅ 프로젝트 방향성과 맞는지
를 함께 확인하는 과정이 반드시 필요하다.
→ Pull Request가 그 확인의 창구 역할을 한다.
```

### 좋은 Pull Request의 조건

```
❌ 정리되지 않은 큰 변경을 한 번에 올리면
   → 리뷰가 어렵고 문제를 찾기도 힘들다

✅ 최대한 작은 단위로 나눠서 올리는 것이 중요
   → 기능 하나, 버그 하나, 수정 하나
```

## 3. GitHub Flow란?

`main` 브랜치를 기준으로 하는 **단순한 브랜치 전략**.  
main 브랜치는 항상 **배포 가능한 상태**를 유지하고,  
모든 작업은 main에서 분기한 **기능 브랜치**에서 이루어진다.

`main`ブランチを基準(きじゅん)とする**単純(たんじゅん)なブランチ戦略(せんりゃく)**。

### 기본 흐름

```
main에서 브랜치 생성
    ↓
작업 (코드 작성)
    ↓
commit / push
    ↓
Pull Request 생성
    ↓
팀원 리뷰
    ↓
main에 merge
    ↓
배포(deploy)
```

### 작업 단위를 작게 나누는 것이 핵심

```
버튼 하나, 기능 하나처럼 작은 단위로 브랜치를 나누면
→ Pull Request도 작아지고
→ 리뷰도 빠르고 간단해진다
```

---

## 4. git-flow vs GitHub Flow

| 구분            | GitHub Flow               | git-flow                                 |
| :-------------- | :------------------------ | :--------------------------------------- |
| 브랜치 수       | 최소 (main + feature)     | 많음 (main, develop, release, hotfix 등) |
| 구조            | 단순                      | 복잡                                     |
| 적합한 프로젝트 | 배포 빈도가 높은 프로젝트 | 명확한 릴리즈 주기가 있는 프로젝트       |

### 왜 GitHub Flow를 쓰는가?

```
배포 빈도가 높고 Pull Request 중심의 리뷰 문화가 있는 조직에서는
복잡한 브랜치 전략보다 단순하고 빠른 GitHub Flow가 더 적합하다.

각 개발자는 기능 브랜치에서 작업을 시작하고
Pull Request를 통해 리뷰를 받은 후 main 브랜치에 merge한다.
```

## 5. 브랜치 이름 규칙

브랜치 이름만 봐도 **어떤 작업인지 파악**할 수 있어야 한다.

```
✅ 좋은 예시
command-study        # git 명령어 학습
pull-request-study   # Pull Request 학습
feature-login        # 로그인 기능
fix-comment-bug      # 댓글 버그 수정

❌ 나쁜 예시
git-command-study    # git/ 폴더 안에 있는데 git을 중복 사용
my-branch            # 어떤 작업인지 불명확
test                 # 너무 모호함
```

## 6. 전체 흐름 한눈에 보기

```bash
# 1. 리모트 리포지토리 복사
git clone https://github.com/user/repo.git

# 2. 기능 브랜치 생성 & 이동
git branch feature-login
git switch feature-login

# 3. 작업 후 상태 확인
git status
git diff

# 4. 스테이지에 올리기
git add .

# 5. 커밋 (의미 있는 단위로!)
git commit -m "feat: 로그인 기능 추가"

# 6. 리모트에 push
git push origin feature-login

# 7. GitHub에서 Pull Request 생성 → 리뷰 → merge

# 8. 팀원 코드 받기
git pull origin main
```

## 🔑 한눈에 보는 키워드

| 명령어              | 설명                                     |
| :------------------ | :--------------------------------------- |
| `git clone`         | 리모트 리포지토리를 로컬에 복사          |
| `git branch`        | 브랜치 생성 / 목록 확인                  |
| `git switch`        | 브랜치 전환                              |
| `git status`        | 워크트리/스테이지 상태 확인              |
| `git add`           | 워크트리 → 스테이지                      |
| `git commit -m`     | 스테이지 → 로컬 리포지토리               |
| `git diff`          | 변경사항 비교                            |
| `git push`          | 로컬 → 리모트(GitHub)                    |
| `git pull`          | 리모트 → 로컬 (fetch + merge)            |
| `git fetch`         | 리모트 변경사항 가져오기만 (merge 안 함) |
| `git merge`         | 다른 브랜치를 현재 브랜치에 합치기       |
| `git log --oneline` | 커밋 이력 한 줄 요약                     |
